# Candy-Spoof

> **⚠️ فقط برای اهداف آموزشی**
> این نرم‌افزار تنها برای اهداف آموزشی، پژوهشی و تست امنیتی مجاز ارائه شده است. کاربران مسئول رعایت تمام قوانین و مقررات مربوطه در حوزه قضایی خود هستند. استفاده غیرمجاز از جعل IP ممکن است غیرقانونی و غیراخلاقی باشد. نویسندگان هیچ مسئولیتی در قبال سوء استفاده از این نرم‌افزار نمی‌پذیرند.

**Candy-Spoof** یک تونل دوطرفه با قابلیت جعل IP است که برای استفاده در شبکه‌های تحت سانسور شدید طراحی شده است. هم کلاینت و هم سرور آدرس IP مبدأ خود را جعل می‌کنند که باعث می‌شود تحلیل ترافیک و مسدودسازی به‌طور قابل توجهی سخت‌تر شود.

## ویژگی‌ها

| ویژگی | توضیحات |
|---------|-------------|
| جعل دوطرفه IP | هم کلاینت و هم سرور از آدرس‌های جعلی مبدأ استفاده می‌کنند |
| استخر IP‌های جعلی | چرخش آدرس‌های مبدأ در هر نشست برای فرار بهتر |
| کانال داده | UDP جعلی با پورت قابل تنظیم |
| کانال کنترل | ICMP Echo Request/Reply جعلی (شبیه ترافیک ping) |
| تحویل قابل اعتماد | ARQ با تکرار انتخابی و ارسال مجدد هر بسته |
| کنترل ازدحام | AIMD شبیه TCP: شروع آهسته، اجتناب از ازدحام، ارسال مجدد سریع، تخمین RTT (RFC 6298) |
| پروکسی SOCKS5 | پروکسی محلی روی پورت 1080 - هر برنامه‌ای را از طریق تونل هدایت کنید |
| اعتبارسنجی لیست سفید | بسته‌ها از IP‌های ناشناخته بی‌صدا رها می‌شوند |
| چندین تونل | تعداد قابل تنظیم تونل‌های موازی مستقل |
| کلید از پیش اشتراک‌گذاری شده | PSK اختیاری برای احراز هویت بسته |

## پیش‌نیازها

- **سیستم‌عامل**: Linux (نیاز به پشتیبانی raw socket)
- **مجوزها**: قابلیت `CAP_NET_RAW` یا دسترسی root
- **کامپایلر**: Rust 1.70 یا بالاتر (ویرایش 2021)

## ساخت از منبع

پروژه را با استفاده از Cargo بسازید:

```bash
cargo build --release
```

فایل‌های اجرایی کامپایل شده در مسیرهای زیر در دسترس خواهند بود:
- کلاینت: `target/release/client`
- سرور: `target/release/server`

## استفاده

### اجرای سرور

سرور را با فایل پیکربندی راه‌اندازی کنید:

```bash
sudo ./target/release/server --config config/server.toml
```

یا در حین توسعه:

```bash
sudo cargo run --bin server -- --config config/server.toml
```

### اجرای کلاینت

کلاینت را با فایل پیکربندی راه‌اندازی کنید:

```bash
sudo ./target/release/client --config config/client.toml
```

پس از راه‌اندازی کلاینت، برنامه‌های خود (مرورگر، curl و غیره) را برای استفاده از پروکسی SOCKS5 در آدرس `127.0.0.1:1080` پیکربندی کنید.

## پیکربندی

| فیلد | کجا | توضیحات |
|-------|-------|-------------|
| `real_ip` | هر دو | آدرس IPv4 واقعی دستگاه شما |
| `peer_real_ip` | هر دو | آدرس IPv4 واقعی همتا |
| `spoofed_ip` | هر دو | IP جعلی مبدأ برای بسته‌های خروجی |
| `peer_spoofed_ip` | هر دو | IP جعلی مبدأ که همتا استفاده می‌کند |
| `spoofed_ip_pool` | هر دو | استخر اختیاری IP‌های مبدأ برای چرخش |
| `data_port` | هر دو | پورت UDP برای کانال داده (باید مطابقت داشته باشد) |
| `icmp_id` | هر دو | شناسه ICMP echo (باید مطابقت داشته باشد) |
| `pre_shared_key` | هر دو | رمز برای احراز هویت بسته (باید مطابقت داشته باشد) |
| `allowed_peers` | هر دو | IP‌های اضافی برای لیست سفید |
| `interface` | هر دو | رابط شبکه (مثلاً `eth0`) |
| `socks5_port` | کلاینت | پورت پروکسی SOCKS5 محلی (پیش‌فرض `1080`) |
| `tunnel_count` | هر دو | تعداد تونل‌های موازی (پیش‌فرض `4`) |
| `mtu` | هر دو | حداکثر بایت بار هر بسته (پیش‌فرض `1380`) |
| `initial_cwnd` | هر دو | پنجره ازدحام اولیه در بسته‌ها (پیش‌فرض `10`) |

**نکته**: پیکربندی کلاینت و سرور باید در فیلدهای حیاتی مانند `data_port`، `icmp_id` و `pre_shared_key` مطابقت داشته باشند تا تونل به درستی کار کند.

برای نمونه‌های کامل پیکربندی به `config/client.toml` و `config/server.toml` مراجعه کنید.

## نحوه کار

```
[برنامه]
     │  TCP (SOCKS5 CONNECT)
     ▼
[پروکسی SOCKS5 – 127.0.0.1:1080]
     │  داده Candy-Spoof (ARQ + CC)
     ▼
[Raw UDP – src=spoofed_client_ip  dst=real_server_ip]
─── شبکه تحت سانسور ──────────────────────────────────
[مدیر تونل – سرور بر اساس tunnel_id جداسازی می‌کند]
     │  TCP
     ▼
[سرور هدف]
     │
     ▼
[Raw UDP – src=spoofed_server_ip  dst=real_client_ip]
─── شبکه تحت سانسور ──────────────────────────────────
[برنامه پاسخ را دریافت می‌کند]
```

**کانال کنترل**: بسته‌های کنترلی (SYN، ACK، NACK، Heartbeat) از پیام‌های ICMP Echo Request/Reply استفاده می‌کنند تا با ترافیک ping معمولی ترکیب شوند و شناسایی را دشوارتر کنند.

## ARQ و کنترل ازدحام

- **ARQ با تکرار انتخابی** – فقط بسته‌های گمشده دوباره ارسال می‌شوند؛ بسته‌های خارج از ترتیب بافر می‌شوند تا شکاف پر شود.
- **شروع آهسته** – پنجره به صورت تصاعدی از `initial_cwnd` تا `ssthresh` رشد می‌کند.
- **اجتناب از ازدحام** – افزایش افزایشی حدود +1 در هر RTT.
- **ارسال مجدد سریع** – با 3 ACK تکراری فعال می‌شود؛ پنجره نصف می‌شود.
- **تایم‌اوت** – پنجره به 1 بازنشانی می‌شود، عقب‌نشینی تصاعدی RTO (تا 60 ثانیه).
- **تخمین RTT** – الگوریتم Jacobson/Karels (RFC 6298).

## ملاحظات امنیتی

- **به حداقل رساندن امتیازات**: در صورت امکان از قابلیت `CAP_NET_RAW` به جای اجرا به عنوان root کامل استفاده کنید
- **محافظت از اعتبارنامه‌ها**: `pre_shared_key` را محرمانه نگه دارید و به طور منظم آن را تغییر دهید
- **انتخاب دقیق IP‌های جعلی**: از آدرس‌های قابل مسیریابی جهانی استفاده کنید که متعلق به شما نیستند؛ آدرس‌های CDN یا DNS عمومی به ترکیب الگوهای ترافیک کمک می‌کنند
- **آگاهی از محدودیت‌ها**: فیلترینگ ورودی (BCP 38) در روترهای بالادست ممکن است بسته‌ها با آدرس‌های مبدأ جعلی را مسدود کند
- **انطباق قانونی**: اطمینان حاصل کنید که مجوز استفاده از این ابزار در محیط شبکه خود را دارید

## نکات عملکرد

- برای موازی‌سازی بیشتر در لینک‌های سریع، `tunnel_count` را افزایش دهید.
- `mtu` را برای مطابقت با path MTU تنظیم کنید (`tracepath` کمک می‌کند).
- `initial_cwnd` (مثلاً `30`) را در لینک‌های با تأخیر کم افزایش دهید تا زمان افزایش شروع آهسته کاهش یابد.
- از `spoofed_ip_pool` با 3-5 آدرس برای توزیع الگوهای ترافیک استفاده کنید.
